# Dynamic Basis Functions and Skinny Tables

## Motivation
Originally, `py-flow` generated full, unrolled Abstract Syntax Trees (ASTs) for every distinct instrument in the portfolio. While robust, this approach encounters scaling limits when pricing portfolios of thousands of swaps across large scenario sets. The overhead of generating, transmitting, and compiling the analytic ASTs for every individual swap's cashflows becomes computationally prohibitive and creates large payloads.

The architecture can decouple the complex non-linear curve math (interpolations, root finding, OIS projection dependencies) from the sheer volume of trades. To price the portfolio dynamically without unrolling exact cashflow math in the DB:
1. **Pre-calculate Sensitivities (Weights):** Outside the scenario engine (in `py-flow`), the exact swap definition (including all stub dates, day count fractions, etc.) is processed to compute a table of weights (sensitivities) relative to the underlying curve knots. 
2. **Standardized Risk Representation:** A given 7.3-year swap is reduced from a complex cashflow schedule into a simple vector of weights on the 7Y and 10Y curve knots.
3. **The Pricing Join:** The `t_swaps` table is transformed into `t_swap_sensitivities` (columns: `Swap_Id`, `Curve_Knot_Id`, `Weight`). Pricing a million swaps across a million scenarios becomes a massive grouped aggregation of a skinny join:
   ```sql
   -- Conceptual aggregation
   NPV = SUM(Swap_Knot_Weight * Scenario_Knot_Value) GROUP BY Scenario_Id, Swap_Id
   ```

This final step is a linear sum of basis functions, but that does not imply it is a linear approximation of the derivative prices. The non-linearities are captured in the basis functions themselves.

Instead of a hard-coded catalog of Basis Functions, the Py-Flow risk engine can **dynamically discover, catalog, and compile** its own library of Basis Functions directly from the `Expr` DAG generated by the `@traceable` execution tracer.

## DAG Pattern Fingerprinting

When `@traceable` evaluates a complex instrument, it produces an explicit mathematical dependency graph (the `Expr` tree), containing `Const`, `Variable` (market knots), `BinOp`, `UnaryOp`, etc.

The compilation engine employs a structural hashing algorithm to abstract and fingerprint sub-graphs:

### Placeholder Substitution
The engine traverses the instrument's DAG and abstracts away specific values:
- All `Variable` nodes (e.g., `IR_USD_10Y`) are replaced with ordered positional placeholders: `X1`, `X2`, etc.
- All `Const` nodes representing trade-specific terms (like Notional, Strike, DayCount) are replaced with ordered parameter placeholders: `p1`, `p2`, `p3`.
- Small integer constants (like `0`, `1`, `2`) used structurally in math are typically preserved (e.g., the `1` in `1 / (1 + r)`).

**Example:**
Specific Swap Sub-tree: `1000000.0 * (1.0 / (1.0 + 0.5 * Variable('IR_USD_10Y')))`
Abstract Pattern: `p1 * (1.0 / (1.0 + p2 * X1))`

The abstracted tree is serialized into a canonical representation (e.g., a normalized Lisp-like prefix string):
`MUL(p1, DIV(1.0, ADD(1.0, MUL(p2, X1))))`

This string is hashed to create a unique `Signature_ID`.

### Dynamic Cataloging and SQL Generation
The engine maintains a central Registry of discovered `Signature_ID`s. For each sub-tree parsed from the portfolio:
- If the `Signature_ID` is **new**:
  1. It assigns a new integer `Component_Type` (e.g., `Type 5`).
  2. It auto-generates the database native code (SQL/Java) for this pattern using the placeholders.
     *Example:* `WHEN 5 THEN p1 * (1.0 / (1.0 + p2 * X1))`
  3. It registers the pattern in the catalog.
- Regardless of whether it is new or existing, the engine extracts the *actual* parameters from the specific trade and writes a simple, flat row to the `t_instrument_components` skinny table:
  `[Instrument_Id: "Swap123", Component_Type: 5, Knot_1: "IR_USD_10Y", p1: 1000000.0, p2: 0.5]`

This gives advantages:
**Zero Boilerplate:** Quants drop newly modeled and exotic derivative classes into py-flow. As long as those models emit standard math via `@traceable`, the compiler automatically discovers the required "basis functions" without manual DB alignment.
**Optimal Compression:** The engine organically finds common denominators. If a bond pricer and a swap pricer both use a standard compounding discount factor, their underlying DAG geometries will hash to the exact same `Signature_ID`, routing both to the exact same SQL SIMD instruction.
**Engine Portability:** Because Py-Flow generates the execution switch-statement strings dynamically, it can instantly pivot to generate raw string code for DuckDB (SQL), Deephaven (Groovy/Java), or C++ execution nodes just by altering the template formatting for the dynamically discovered patterns.

### Array Unrolling and Ungrouping
Within the analytical engine (Deephaven or DuckDB), the evaluated scalar columns for the discrete tenors are first gathered into array columns per `Scenario_Id`. We then apply an `ungroup()` (or unpivot) operation to explode the arrays into a narrow, "skinny" table format.

Instead of generating columns for each market data point, market quotes and curve knots must be represented as rows. The scenario table should serve the coordinates of market data as primary keys in a skinny column format:
- `Scenario_Id`
- `MarketData_Id` (or `Curve_Knot_Id`)
- `Value` (e.g., rate, shift, or simulated quote)

This gives advantages:
**O(1) AST Compilation:** The AST generation time is strictly determined by the number of discrete tenors (e.g., 10), entirely independent of the portfolio size.
**High-Performance Scaling:** Offloads the heavy lifting of portfolio evaluation to Deephaven/DuckDB, leveraging their highly optimized, vectorized column math and relational JOINS.
**Reduced Data Payload:** Minimizes code-generation size and the data movement required between Py-Flow and the execution server.
**Memory Efficiency:** "Skinny table" joins exhibit simpler memory access patterns, are easier to partition/shard, and compress exceptionally well compared to extremely wide single-row schemas.

## Scaling and Performance Benchmarks

The `scripts/scaling_ir_swap.py` suite benchmarks this decoupled representation against legacy unrolled executions. By leveraging fast vector execution in DuckDB/Deephaven directly on Skinny table components, we achieve structural latency reductions.

### Micro-Portfolio (10 Swaps)
To understand why advanced architectures are necessary, consider the baseline execution time to price and calculate risks on merely 10 fully featured Interest Rate Swaps. 

*(Note: Data for portfolios ≤ 100 instruments are looped and averaged over 10 iterations internally to normalize away static engine startup/query compilation jitter, leaving only true marginal processing times.)*

| Engine | NPV (ms) | Per-Instr Risk (ms) | Port-Total Risk (ms) | Atoms |
|---|---|---|---|---|
| Python Float (Raw) | 124.5 | 1949.5 | 695.3 | - |
| Python Expr (DAG) | 3.7 | 15.9 | 20.7 | - |
| NumPy Vectorized | 10.1 | 10.1 | 10.1 | - |
| DuckDB Skinny | 6.3 | 4.9 | 4.1 | 355 |
| Deephaven Skinny | 17.0 | 11.8 | 10.4 | 355 |

*At this granular scale, DB engines and NumPy array traversals are nearly instantaneous, hovering in the micro-to-millisecond margins. However, `Python Float` explicitly breaks down, measuring nearly 2.0 seconds for per-instrument risk (`bump-and-reprice`) because running numerical shifts on raw numerical float solvers scales exponentially poorly compared to Exact analytical Jacobians.*

### Institutional Portfolio (2,000 Swaps)
At 2,000 swaps, the legacy pure Python fallback is permanently skipped (it would theoretically require roughly ~25 minutes). This highlights the scaling capabilities of memory-mapped analytical execution. The fix for Deephaven enables Docker memory-mapped Parquet transfer, allowing it to easily clear 2,000 swaps. 

| Engine | NPV (ms) | Per-Instr Risk (ms) | Port-Total Risk (ms) | Atoms |
|---|---|---|---|---|
| NumPy Vectorized | 2635.7 | 2635.7 | 2635.7 | - |
| Deephaven Skinny | 168.3 | 136.0 | 143.7 | 96,981 |
| DuckDB Skinny | 25.4 | 21.5 | 18.7 | 96,981 |

*At 2,000 trades, DuckDB decisively sweeps the benchmark natively! Because relational DB engines effortlessly vectorize CASE evaluations across 96,000 atomic dependencies simultaneously on SIMD layers, it achieves an incredible ~100x acceleration (~21ms vs ~2.6s) over NumPy unrolled vectors.*

### Massive Scale (10,000 Swaps)
At massive institutional scales involving tens of thousands of swaps, pure Python frameworks (like `Python Symbolic`) simply exhaust available memory footprint and computation time. However, database layer translation continues to scale efficiently.

*(Total Active Evaluated Dependency Atoms: ~482,255)*

| Engine | NPV (ms) | Per-Instr Risk (ms) | Port-Total Risk (ms) | Peak Process Memory |
|---|---|---|---|---|
| NumPy Vectorized | 13,402.0 | 13,402.0 | 13,402.0 | ~873 MB |
| Deephaven Skinny | 695.8 | 589.0 | 599.6 | ~2,860 MB |
| DuckDB Skinny | 29.0 | 29.8 | 23.9 | ~2,830 MB |

*Note: Generating nearly a half-million dependency atoms intrinsically maps ~2.8 GB natively for DuckDB and Deephaven to safely hold query evaluation blocks, but the actual calculation of those 10,000 swaps mathematically bridges roughly a phenomenal 450x faster acceleration in C++ SQL than native NumPy unrolling.*
