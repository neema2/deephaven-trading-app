# Dynamic Basis Function Extraction

## Abstract
In a heavily parameterized portfolio, explicitly coding every possible standard and non-standard mathematical cashflow component into a finite catalog of SQL/Deephaven `CASE` statements is time-consuming and fragile. 

Instead of a hard-coded catalog of Basis Functions, the Py-Flow risk engine can **dynamically discover, catalog, and compile** its own library of Basis Functions directly from the `Expr` DAG generated by the `@computed_expr` execution tracer.

## The Process: DAG Pattern Fingerprinting

When `@computed_expr` evaluates a complex instrument, it produces an explicit mathematical dependency graph (the `Expr` tree), containing `Const`, `Variable` (market knots), `BinOp`, `UnaryOp`, etc.

The compilation engine employs a structural hashing algorithm to abstract and fingerprint sub-graphs:

### 1. Placeholder Substitution
The engine traverses the instrument's DAG and abstracts away specific values:
- All `Variable` nodes (e.g., `IR_USD_10Y`) are replaced with ordered positional placeholders: `X1`, `X2`, etc.
- All `Const` nodes representing trade-specific terms (like Notional, Strike, DayCount) are replaced with ordered parameter placeholders: `p1`, `p2`, `p3`.
- Small integer constants (like `0`, `1`, `2`) used structurally in math are typically preserved (e.g., the `1` in `1 / (1 + r)`).

**Example:**
Specific Swap Sub-tree: `1000000.0 * (1.0 / (1.0 + 0.5 * Variable('IR_USD_10Y')))`
Abstract Pattern: `p1 * (1.0 / (1.0 + p2 * X1))`

### 2. Structural Fingerprinting (Hashing)
The abstracted tree is serialized into a canonical representation (e.g., a normalized Lisp-like prefix string):
`MUL(p1, DIV(1.0, ADD(1.0, MUL(p2, X1))))`

This string is hashed to create a unique `Signature_ID`.

### 3. Dynamic Cataloging and SQL Generation
The engine maintains a central Registry of discovered `Signature_ID`s. For each sub-tree parsed from the portfolio:
- If the `Signature_ID` is **new**:
  1. It assigns a new integer `Component_Type` (e.g., `Type 5`).
  2. It auto-generates the database native code (SQL/Java) for this pattern using the placeholders.
     *Example:* `WHEN 5 THEN p1 * (1.0 / (1.0 + p2 * X1))`
  3. It registers the pattern in the catalog.
- Regardless of whether it is new or existing, the engine extracts the *actual* parameters from the specific trade and writes a simple, flat row to the `t_instrument_components` skinny table:
  `[Instrument_Id: "Swap123", Component_Type: 5, Knot_1: "IR_USD_10Y", p1: 1000000.0, p2: 0.5]`

## Advantages
1. **Zero Boilerplate:** Quants drop newly modeled and exotic derivative classes into py-flow. As long as those models emit standard math via `@computed_expr`, the compiler automatically discovers the required "basis functions" without manual DB alignment.
2. **Optimal Compression:** The engine organically finds common denominators. If a bond pricer and a swap pricer both use a standard compounding discount factor, their underlying DAG geometries will hash to the exact same `Signature_ID`, routing both to the exact same SQL SIMD instruction.
3. **Engine Portability:** Because Py-Flow generates the execution switch-statement strings dynamically, it can instantly pivot to generate raw string code for DuckDB (SQL), Deephaven (Groovy/Java), or C++ execution nodes just by altering the template formatting for the dynamically discovered patterns.
